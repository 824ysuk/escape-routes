# Contributing

事例追加・既存事例の改善 PR を歓迎します。

## 新規事例の追加

1. `cases/NN-short-name.md` を作成（番号は連番）
2. 以下の構造で書く:

```markdown
# 事例 N: <タイトル>

## 何が起きるか
<直接ルートで詰まる症状の具体描写>

## 代替手段（比較表）
| 手段 | 概要 | 再現性 / 用途 |
|---|---|---|
| A. ... | ... | ... |

## 実装例

### A. <手段名>

**前提 / install**: install コマンド、必要なツール、認証情報

**コード**:

\`\`\`bash
<コピペで動く完全なコード>
\`\`\`

**期待出力**: 実行後に何が見えるか、成功 / 失敗の見分け方

**ハマりポイント**: OS / バージョン / 権限 / 制限事項

### B. ... (以下同じ構造)

## 他手段を選ぶ条件
- B: <どんなとき B が合うか>
- C: <どんなとき C が合うか>

## 補足
<一行で全体観察>
```

3. README.md の事例一覧表に行を追加
4. 共通パターンの該当事例番号を更新

## 既存事例の改善

- 誤字 / バージョン更新: 直接 PR
- 新しい代替手段の追加: 比較表に追記 + 実装例セクション追加
- ハマりポイントの追加: 該当事例の「ハマりポイント」項目に追記（過去事例も歓迎）

## 各代替手段の品質基準

- **前提 / install**: 必要なツールの install コマンドと version 要件が読み取れる
- **コード**: プレースホルダー（`YOUR_KEY` 等）はあって良いが、文法・引数・パイプは完全
- **期待出力**: 成功 / 失敗を目で見て判別できるレベル
- **ハマりポイント**: 実運用で典型的に遭遇する罠を網羅、セキュリティ警告（PII / token 漏洩等）が必要箇所に

## トーン

観察記録モード（事実 / 観察 / 参照元を並べる）。「ぜひ」「〜こそ」「重要です」等の説得語彙は使わない。
